# MCW Launcher Core Public API Reference

> Target version: **MCW Launcher v0.4.x**
>
> Audience: GUI, CLI, and other frontend developers.
>
> This document describes the public Core APIs that a frontend may call. It is intentionally framework-neutral.
>
> This was generated by ChatGPT 5.2.
---

## 1. Scope

A frontend may:

- Load version information.
- Create and manage instances.
- Load and save instance settings.
- Manage accounts.
- Authenticate offline accounts.
- Launch Minecraft.
- Receive structured progress events.

A frontend should not:

- Read or write launcher JSON files directly.
- Access the account database directly.
- Download Minecraft files itself.
- Build classpaths or Java commands.
- Select Java runtimes itself.
- Extract native libraries.
- Call methods whose names begin with `_`.

# 3. Version Manifest API

## 3.1. `VersionManifestManager.get()`

### Signature

```python
VersionManifestManager.get() -> list[VersionManifest]
```

### Purpose

Load the Minecraft version manifest.

### Returns

A list of `VersionManifest` models.

### Common fields

```python
version.id
version.type
version.url
version.release_time
```

### Typical usage

```python
versions = VersionManifestManager.get()

release_ids = [
    version.id
    for version in versions
    if version.type == "release"
]
```

### Threading

May perform a network request. Run it outside the GUI thread.

### Notes

- The returned list may include releases, snapshots, old alpha versions, and old beta versions.
- A frontend may filter by `version.type`.
- Do not download the manifest manually.

---

## 3.2. `VersionManifestManager.latest_version()`

### Signature

```python
VersionManifestManager.latest_version(is_snapshot: bool = False) -> str
```

### Parameters

| Name | Type | Description |
|---|---|---|
| `is_snapshot` | `bool` | `False` for latest release, `True` for latest snapshot |

### Returns

The latest version ID.

### Example

```python
latest_release = VersionManifestManager.latest_version()
latest_snapshot = VersionManifestManager.latest_version(is_snapshot=True)
```

### Threading

May depend on manifest loading and should be treated as potentially blocking.

### Notes

The current implementation may return an empty string when the manifest is unavailable.

# 4. Version Metadata API

## 4.1. `VersionManager.load()`

### Signature

```python
VersionManager.load(id: str = VersionManifestManager.latest_version()) -> Version
```

### Parameters

| Name | Type | Description |
|---|---|---|
| `id` | `str` | Minecraft version ID |

### Returns

A `Version` model.

### Example

```python
version = VersionManager.load("1.20.1")
```

### Threading

Performs network and filesystem work. Run it on a worker thread.

### Raises

Possible failures include:

- Version ID not found.
- Manifest unavailable.
- Version metadata download failure.
- Invalid or incomplete metadata.

### Frontend notes

Use this API when:

- Creating a new instance.
- Showing detailed version information.
- Displaying the required Java major version.

Do not call private methods such as `_parse_version()` from a frontend.

---

## 4.2. `Version` model

### Common fields

```python
version.id
version.type
version.path
version.main_class
version.java_version
version.assets
version.asset_index
version.arguments
version.minecraft_arguments
version.libraries
version.downloads
version.raw_json
```

### Field notes

| Field | Type | Meaning |
|---|---|---|
| `id` | `str` | Minecraft version ID |
| `type` | `str` | Release type such as `release`, `snapshot`, `old_alpha`, or `old_beta` |
| `path` | `Path` | Cached version metadata path |
| `main_class` | `str` | Java main class |
| `java_version` | `dict` | Required Java metadata |
| `assets` | `str` | Asset index name |
| `asset_index` | `dict` | Asset index download metadata |
| `arguments` | `dict \| None` | Modern structured JVM/game arguments |
| `minecraft_arguments` | `str \| None` | Legacy launch argument string |
| `libraries` | `list[dict]` | Library metadata |
| `downloads` | `dict` | Client/server download metadata |
| `raw_json` | `dict` | Full parsed metadata |

### Compatibility note

Modern versions usually use:

```text
arguments.jvm
arguments.game
```

Legacy versions may use:

```text
minecraftArguments
```

A frontend must not parse or choose between these formats. The launch pipeline handles them automatically.

# 5. Instance API

## 5.1. `InstanceManager.list_instances()`

### Signature

```python
InstanceManager.list_instances() -> list[Instance]
```

### Purpose

Return all registered instances.

### Returns

A list of `Instance` models.

### Threading

Usually fast. It may run on the GUI thread, but a worker is safer during startup if the instance list is large.

### Example

```python
instances = InstanceManager.list_instances()

for instance in instances:
    print(instance.name, instance.version_id)
```

### Frontend notes

Refresh the displayed list after:

- Create.
- Rename.
- Clone.
- Delete.
- Import.

---

## 5.2. `InstanceManager.create()`

### Signature

```python
InstanceManager.create(name: str, version: Version, mod_loader: tuple[str, str] = ("vanilla", "-1")) -> Instance
```

### Parameters

| Name | Type | Description |
|---|---|---|
| `name` | `str` | New instance name |
| `version` | `Version` | Loaded Minecraft version model |
| `mod_loader` | `tuple[str, str]` | Loader name and version |

### Returns

The created `Instance`.

### Example

```python
version = VersionManager.load("1.20.1")
instance = InstanceManager.create(name="My Instance", version=version)
```

### Threading

Performs filesystem writes. Worker thread recommended.

### Raises

Common failures:

- Instance name already exists.
- Invalid filesystem name.
- Directory creation failure.
- Metadata write failure.

### Frontend notes

After success:

- Refresh the instance list.
- Select the newly created instance.
- Load its settings if needed.

## 5.6. `InstanceManager.clone()`

### Signature

```python
InstanceManager.clone(source_name: str, new_name: str, include_saves: bool = False) -> Instance
```

### Parameters

| Name | Type | Description |
|---|---|---|
| `source_name` | `str` | Existing instance name |
| `new_name` | `str` | Clone name |
| `include_saves` | `bool` | Whether to copy worlds |

### Returns

The cloned `Instance`.

### Threading

Run on a worker thread. Large worlds may take a long time.

### Notes

When `include_saves=False`, large or disposable directories may be excluded, including:

- `saves`
- `logs`
- `crash-reports`

### Frontend notes

Show an explicit option before copying saves because world folders may be very large.

---

## 6.2. `SettingsManager.save()`

### Signature

```python
SettingsManager.save(instance: Instance, settings: InstanceSettings) -> None
```

### Purpose

Persist the complete settings model.

### Threading

Usually fast, but it performs filesystem writes.

### Example

```python
settings = SettingsManager.load(instance)
settings.min_memory = 1024
settings.max_memory = 4096
SettingsManager.save(instance, settings)
```

### Frontend validation

Before saving:

- `min_memory > 0`
- `max_memory >= min_memory`
- `width > 0`
- `height > 0`
- Custom Java path exists, when set

---

## 6.3. Additional update methods

Depending on the current implementation, the manager may expose:

```python

SettingsManager.update_window(...)
SettingsManager.update_jvm_arguments(...)
SettingsManager.update_game_arguments(...)
```

Use these only when present in the current public implementation. `load()` plus `save()` remains the simplest stable workflow.

# 8. Offline Authentication API

## 8.1. `OfflineAuthentication.uuid_generator()`

### Signature

```python
OfflineAuthentication.uuid_generator(username: str) -> str
```

### Purpose

Generate a stable Minecraft offline UUID.

### Threading

Safe on the GUI thread.

### Example

```python
player_uuid = OfflineAuthentication.uuid_generator("Steve")
```

### Notes

The result is:

- Stable for the same username.
- Case-sensitive.
- Compatible with Minecraft offline UUID behavior.

---

## 8.2. `OfflineAuthentication.authenticate()`

### Signature

```python
OfflineAuthentication.authenticate(account: Account) -> Authentication
```

### Returns

An `Authentication` model ready for `MinecraftExecutor.run()`.

### Threading

Safe on the GUI thread.

### Example

```python
authentication = OfflineAuthentication.authenticate(account)
```

---

## 8.3. Temporary offline account example

A frontend may create an in-memory account for a temporary session:

```python
import uuid

account = Account(
    account_id=str(uuid.uuid4()),
    account_type=AccountSource.OFFLINE,
    username="Steve",
    uuid=OfflineAuthentication.uuid_generator("Steve"),
)

authentication = OfflineAuthentication.authenticate(account)
```

For a full account screen, prefer persisted accounts through `AccountManager`.

# 9. Launch API

## 9.1. `MinecraftExecutor.run()`

### Signature

```python
MinecraftExecutor.run(
    instance: Instance,
    authentication: Authentication,
    account: Account,
    on_progress: ProgressCallback | None = None,
    debug_mode: bool = False,
) -> dict
```

### Parameters

| Name | Type | Description |
|---|---|---|
| `instance` | `Instance` | Instance to launch |
| `authentication` | `Authentication` | Authentication data |
| `account` | `Account` | Account model |
| `on_progress` | `Callable[[ProgressEvent], None] \| None` | Progress callback |
| `debug_mode` | `bool` | Enable extra debug behavior |

### Returns

The current implementation returns:

```python
{
    "javaPath": Path,
    "minecraftJavaMajorVersion": int,
    "minecraftVersion": str,
}
```

### Threading

Must run on a worker thread.

### Example

```python
result = MinecraftExecutor.run(
    instance=instance,
    authentication=authentication,
    account=account,
    on_progress=on_progress,
)
```

### Internal responsibilities

The executor owns the complete launch pipeline:

```text
Load version metadata
Download client
Download libraries
Download asset index
Download assets
Load settings
Build context
Build arguments
Build command
Select Java
Launch Minecraft
```

A frontend must not call these internal steps separately.

### Compatibility

The executor handles both:

- Modern structured arguments.
- Legacy `minecraftArguments`.

The frontend does not need version-specific launch logic.

# 10. Progress API

## 10.1. Progress callback

### Signature

```python
def on_progress(event: ProgressEvent) -> None:
    ...
```

The callback may be invoked from the worker thread running the executor.

---

## 10.2. `ProgressEvent` model

### Common fields

```python
event.stage
event.message
event.current
event.total
event.unit
event.fraction
event.percentage
event.is_determinate
```

### Field meanings

| Field | Type | Meaning |
|---|---|---|
| `stage` | `ProgressStage` | Current pipeline stage |
| `message` | `str` | Human-readable status |
| `current` | `int \| None` | Current amount |
| `total` | `int \| None` | Total amount |
| `unit` | `str \| None` | Unit such as files or bytes |
| `fraction` | `float \| None` | Value from 0.0 to 1.0 |
| `percentage` | `float \| None` | Value from 0 to 100 |
| `is_determinate` | `bool` | Whether total progress is known |

### Frontend behavior

If `is_determinate` is true:

- Show percentage.
- Show current and total.
- Use a determinate progress bar.

If false:

- Show the message and stage.
- Use an indeterminate progress indicator.

---

## 10.3. `ProgressStage`

Current stages include:

```python
ProgressStage.PREPARING
ProgressStage.LOADING_VERSION
ProgressStage.DOWNLOADING_CLIENT
ProgressStage.DOWNLOADING_LIBRARIES
ProgressStage.DOWNLOADING_ASSET_INDEX
ProgressStage.DOWNLOADING_ASSETS
ProgressStage.BUILDING_CONTEXT
ProgressStage.BUILDING_COMMAND
ProgressStage.SELECTING_JAVA
ProgressStage.LAUNCHING
ProgressStage.FINISHED
```

| Stage | Meaning |
|---|---|
| `PREPARING` | Initialize the pipeline |
| `LOADING_VERSION` | Load and parse version metadata |
| `DOWNLOADING_CLIENT` | Verify or download client JAR |
| `DOWNLOADING_LIBRARIES` | Verify or download libraries |
| `DOWNLOADING_ASSET_INDEX` | Download asset index metadata |
| `DOWNLOADING_ASSETS` | Verify or download asset objects |
| `BUILDING_CONTEXT` | Build placeholder context |
| `BUILDING_COMMAND` | Build final command |
| `SELECTING_JAVA` | Choose Java runtime |
| `LAUNCHING` | Start the Java process |
| `FINISHED` | Process creation completed |

`FINISHED` means the Minecraft process was created. It does not mean the game process has exited.

# 13. Error Contract

Current public APIs commonly raise standard Python exceptions or `RuntimeError`.

Potential categories include:

- `RuntimeError`
- `FileNotFoundError`
- `PermissionError`
- `OSError`
- Network-related request errors
- Invalid package or metadata errors

Frontend requirements:

- Catch errors at the public API boundary.
- Show concise user-facing messages.
- Keep technical details in an activity/debug log.
- Never silently swallow unexpected exceptions.

# 16. Frontend Checklist

- [ ] Uses only public Core APIs.
- [ ] Does not read JSON or SQLite directly.
- [ ] Runs network and launch tasks outside the UI thread.
- [ ] Displays `ProgressEvent` instead of parsing logs.
- [ ] Refreshes instances after create, rename, clone, delete, or import.
- [ ] Confirms destructive actions.
- [ ] Validates user input before calling Core.
- [ ] Prevents duplicate launch tasks.
- [ ] Handles errors visibly.
- [ ] Does not contain version-specific launch logic.
